## 1. 這個系統做什麼

DCT_data_import 是一支 **SDK-style .NET 8 Console 應用程式**(`net8.0-windows`),負責把半導體 **DCT(Direct Circuit Test)測試資料**從 FTP / 檔案系統自動匯入 MySQL 資料庫。

核心流程:**從 FTP 或 LocalImportRoot 取得測試結果檔(big5 編碼)→ 依檔案格式解析 → 驗證 → 批次寫入 MySQL → 成功時清理來源檔並寫 success log → 以旗標(check_status)判定該批是否完整匯入 → 失敗時回寫 mail 旗標並寄通知信**。

程式啟動後進入**無限迴圈**,常駐三條工作執行緒輪詢待匯入資料,屬於長時間運行的背景匯入服務(非一次性 ETL 批次工具)。

## 4. 進入點與三執行緒模型(Program.cs)

`Program.cs`(約 670 行)的 public 介面:`Main` / `GetEnvironment` / `GetLocalIPAddress` / `SendMailModel` / `ImportTesterMode` / `ImportUiStatusMode` / `ImportTsmcMode`。

### 4.1 三條常駐工作執行緒

`Main` 啟動後進入**無限迴圈**,常駐三條工作執行緒(**不是兩條**;舊報告誤述為「Tester / TSMC 兩模式選擇」):

| 執行緒 | 方法 | 職責 |
|---|---|---|
| Tester 模式 | `ImportTesterMode` | 處理 RawData / Tester / FailPin / RecoveryRate / MultiSpec 等主要測試資料,並計算 check_status |
| UiStatus 模式 | `ImportUiStatusMode` | 獨立處理 UI 狀態(寫 `UI_status` 表,**不在 check_status 旗標內**) |
| TSMC 模式 | `ImportTsmcMode` | 處理 TSMC IEDA 對接(固定寬度解析、寫 ieda_title/ieda_content) |

### 4.2 執行緒自我復活

迴圈會偵測死掉的執行緒並透過 `RestartWorker` 重啟:`Thread.Interrupt()` → `new Thread(...)` → `Start()`。`Thread.Abort()` 在 net5+ 已移除,目前只保留在註解中說明為何不再呼叫。

> ⚠️ **`Thread.Sleep(432000)` 的註解寫「2H」但實際是 7.2 分鐘**(432000 **毫秒** = 432 秒)。這是單位陷阱,迴圈實際每約 7.2 分鐘輪詢一次,非每 2 小時。

### 4.3 其他注意事項

- `ImportTsmcMode` **不讀取 DB 的 DB_Key 清單**(與 Tester 模式不同,直接掃 FTP 目錄)。
- `Main` 啟動時只輸出 HOST/USER 與 PASSWORD set/unset,不印 PASSWORD 明文;`App.config` 內既有 DB/FTP secret 仍是 S1 風險。

## 5. 環境與資料來源切換機制

**靠本機 IPv4 硬編碼判定,不是用 build configuration**:

- `GetLocalIPAddress()` 取本機 IPv4
- `productionIps = {"10.16.92.67", "10.16.92.68"}` → 命中即為 **Prod**,否則 **Dev**(`GetEnvironment`)
- 連線參數鍵由字串串接組成:`${Environment}Host`、`${Environment}UserName`… 對應 App.config 的 `DevHost`/`ProdHost` 等

| 環境 | DB | FTP 根目錄 |
|---|---|---|
| Dev | configured Dev host/user/db | `/DCT_Log/DCT_DB_DATA_Dev/` |
| Prod | configured Prod host/user/db | `/DCT_Log/DCT_DB_DATA/` |

> Dev / Prod 共用同一 database 名稱 `dct`,差別只在 Host。**Release build 不會自動指向 Prod**——環境選擇是 runtime 邏輯,純看執行該程式的機器 IP。FTP 來源主機(`10.16.92.67`)與 Prod MySQL 同一台內網主機。

### 5.1 資料來源切換(FTP / Local)

`ImportFileSourceFactory.Create()` 依 App.config 的 `ImportSource` 建立檔案來源:

- 未設定或不是 `Local` → `FtpImportFileSource`,路徑仍依 Dev/Prod 切到 `/DCT_Log/DCT_DB_DATA_Dev/` 或 `/DCT_Log/DCT_DB_DATA/`
- `ImportSource=Local` → `LocalImportFileSource`,從 `LocalImportRoot` 底下讀同一套相對路徑
- Local 成功後依 `LocalSuccessAction` 決定 `Delete`(預設)或移到 `Imported/`;失敗則移到對應 error path

## 7. CheckStatus 位元旗標模型(GROUND TRUTH)

這是判定「一筆 DB_Key 是否完整匯入」的核心機制。**舊報告的位元順序完全相反且多加了一位 UiStatus,本版為正確版本。**

### 7.1 4-bit 旗標(範圍 0–15)

| bit | 權重 | 子項 | 對應匯入 |
|---|---|---|---|
| bit0 | 1 | **FailPin** | 失效針腳匯入成功 |
| bit1 | 2 | **RawData / TestResult** | 原始測試結果匯入成功 |
| bit2 | 4 | **Tester** | 機台狀態匯入成功 |
| bit3 | 8 | **RecoveryRate** | 回收率匯入成功 |

> **UiStatus 不在這個旗標內**(它寫獨立的 `db_key_ui_status` 表)。舊報告的 bit4=UiStatus、CheckStatus=31 範例皆為錯誤。

### 7.2 判定公式(DbAccess.cs)

```
importResult = 8 × recoveryRate + 4 × tester + 2 × testResult + 1 × failPin
```

- 每個子項匯入成功該位元置 1
- **只有 `importResult == db_key.check_status` 時**,才把 `db_key.import_status` 設為 `1`(完整匯入)
- `check_status` 是該 DB_Key **預期**要匯入的子項組合;不同產品/流程的預期值不同
- 權重 8(RecoveryRate)是近期才加入的子項(commit `479f29e`,連帶新增 `recovery_rate` 表欄位)

## 8. 資料庫存取層

### 8.1 DbAccess.cs(約 381 行)

### 8.2 DatabaseService.cs
- `ExecuteQuery`:primary SELECT API,回傳 `DbQueryResult`(`JArray Data` + `Error`)
- `ExecuteCommand`:primary INSERT/UPDATE/DELETE API,回傳 `DbCommandResult`(`AffectedRows` + `InsertId` + `Error`)
- `CheckDatabaseAndTableExists`:透過 `INFORMATION_SCHEMA.SCHEMATA` / `INFORMATION_SCHEMA.TABLES` **只做存在性檢查**(commit `6e0ead4`)

  - 此方法由 FileProcess 在各 `Import*` / `Delete*` 執行 DELETE 前呼叫(確認標的存在)
- `GetSafeErrorMessage`:回傳時不含 StackTrace

### 8.3 DBmysql.cs
- Dapper `Query` / `Execute`;每次呼叫 `new MySqlConnection`;S2 後支援 `parameters`
- `ExecuteQuery`:SELECT path,回 `DbQueryResult`
- `ExecuteCommand`:non-query path,走 transaction + `Execute` + INSERT 時讀 `LAST_INSERT_ID()`,回 `DbCommandResult`
- `FilterSqlCommand`:以 regex 過濾雙引號 pattern(commit `0a9c2ae`,見 §11.4)
- `MySqlConnectionManager`:static `volatile` + `lock` **單次初始化**全程共用;連線字串 `Pooling Min5/Max100`;驗證除密碼外所有參數

### 8.4 DbObject.cs
- `DbKeyObject`:4 個多載建構式
- `DbSqlRequest`:輸入 DTO,包含 `Query` + `Parameters`
- `DbQueryResult`:SELECT typed result,包含 `JArray Data` + `string Error`
- `DbCommandResult`:command typed result,包含 `AffectedRows` + `InsertId` + `string Error`
- ⚠️ **回應仍以 `Error` 字串傳遞錯誤而非拋例外**:呼叫端必須主動 `string.IsNullOrEmpty(response.Error)` 檢查。typed split 已降低 SELECT 與 command 語意混淆。

> 📌 **命名更新**:DB SQL request DTO 已改為 `DbSqlRequest`。**現況是 Dapper 直連 MySQL,沒有外部 HTTP API Service**(舊報告所述「HTTP API Service 外部系統」已不存在)。原 `ExecuteInsertWithAPI` 方法、API 連線設定與「Web API body」註解已於近期清理；Task 5 後舊 DB result adapter/API surface 也已移除。

- **「事務回滾」實為 best-effort cascade DELETE**(見 §10)

## 10. 錯誤回滾的真相:best-effort cascade DELETE

> **舊報告宣稱的「事務回滾」並不存在真實 DB transaction 保護整個匯入單元。**實際是失敗時以串接的 DELETE 嘗試清掉已寫入的資料,有兩個已知缺口:

1. **lots_result 失敗 → 殘留 orphan**:`lots_info` 與 `lots_statistic` 已先寫入,若 `lots_result` 失敗,DELETE 不一定清得乾淨,留下孤兒資料。
2. **MultiSpec 多 site 失敗 → 部分匯入**:多 site 流程中後段 site 失敗,**前段已成功的 site 不會被回滾**。

也**沒有通用的重試 / 狀態復原機制**。唯一的重試是 `TsmcIeda.GetNetNameList` 的一次性遞迴重試。狀態的「復原」是靠:`db_key.import_status` 標記 + 失敗回寫 `mail=1` + 7.2 分鐘迴圈下次重新撈未完成的 DB_Key 重做。

## 12. 同步模型:明確同步阻塞

> P2 後已移除 fake async 外殼。系統仍是同步阻塞 pipeline,只是現在方法簽名與實際行為一致:

- importer 回傳 `ImportResult`,由 `Program.cs` 工作執行緒直接呼叫
- primary DB API 是同步 `DatabaseService.ExecuteQuery` / `ExecuteCommand`;Task 5 後 DB result surface 已是 typed-only。
- `DBmysql` 仍使用同步 `MySqlConnection.Open` + Dapper `Query` / `Execute`
- 這不是「真 async」改造;若未來要提升 I/O 併發,需另行設計完整 async DB/FTP path

`WriteToLog` 也刻意改回同步 `StreamWriter.WriteLine`(註解說明:避免 `WriteLineAsync` fire-and-forget 丟日誌)。
