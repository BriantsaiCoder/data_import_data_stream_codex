1 系統概觀

一支長時間運行的背景服務,把半導體 DCT(Direct Circuit Test)測試結果檔從 FTP 或 LocalImportRoot 自動匯入 MySQL。啟動後進入無限迴圈,常駐三條工作執行緒,每約 7.2 分鐘輪詢一次待匯入資料。

🎯 核心目的

FTP / Local 測試檔 → 解析 → 驗證 → 批次寫入 MySQL

🔁 運行模式

無限迴圈 + 三常駐執行緒(死亡自我復活)

🚩 完整性判定

4-bit check_status 旗標,逐子項加權比對

📧 失敗處置

回寫 mail=1 + 下輪重做 + 寄通知信

資料來源由 ImportSource 決定:預設 FTP;Local 時從 LocalImportRoot 讀同一套相對路徑,成功後 Delete 或 Archive,並寫入 success log。

5 資料流

單批 DB_Key 的處理流程。

flowchart LR
    A["取得檔案<br/>FTP/Local big5"] --> B["解析<br/>依格式選處理器"]
    B --> C["驗證<br/>Compare* 子集 + 統計清洗"]
    C --> D["批次寫入<br/>Dapper 參數化 + 批次切割"]
    D --> E{"importResult<br/>== check_status ?"}
    E -->|相符| F["import_status=1<br/>清源檔 + 成功 log"]
    E -->|不符| G["import_status=2 + mail=1<br/>寫 mail_temp + 下輪重做"]
成功時:清理來源檔並寫 DCT_data_import_Success_Log_yyyy_MM_dd.txt。失敗時:回寫 db_key.mail=1 + 寫本機 mail_temp.txt → 7.2 分鐘迴圈下輪重撈未完成的 DB_Key 重做。沒有真實 DB transaction(見 ⑪)。
影子驗證模式(DryRun):App.config 設 DryRun=true(預設 false、缺 key 亦 false)時整條 ETL 變唯讀 —— DB 寫入(含 cascade delete)、FTP 刪/改名、Local 搬檔、寄信全部轉 no-op,只解析/比對並保留 mail_temp.txt 供比對;SELECT 仍照常。供 .NET 8 cutover 影子試跑與資料污染回滾防護(Common/RuntimeMode.cs · DBmysql.cs:98 · EmailModels.cs:25)。

6 三執行緒模型 舊報告誤述為兩模式

Main 無限迴圈常駐三條工作執行緒。迴圈偵測死亡執行緒並透過 RestartWorker 以 Interrupt → new → Start 重啟;現行程式不再呼叫 Thread.Abort()。

ImportTesterMode
Tester 模式

主要測試資料:RawData / Tester / FailPin / RecoveryRate / MultiSpec,並計算 check_status。

ImportUiStatusMode
UiStatus 模式

獨立處理 UI 狀態,寫 ui_status 表。不在 check_status 旗標內。

ImportTsmcMode
TSMC 模式

TSMC IEDA 對接,固定寬度解析。不讀 DB 的 DB_Key 清單,直接掃 FTP 目錄。

🐞 單位陷阱:Thread.Sleep(432000) 註解寫「2H」,但 432000 毫秒 = 432 秒 = 7.2 分鐘。實際輪詢週期是 7.2 分鐘,不是 2 小時。

7 七個格式處理器

ImportData.cs 是基底類別(big5 / FTP / 路徑);以下 7 個處理器繼承它。舊報告誤列的 *Refactored.cs 全部不存在。各處理器行數與方法見 docs/codebase/STRUCTURE.md。

RawData

ReadAndImport/RawData.cs

表:lots_info / lots_statistic / lots_result

result 代碼:0=檔案不存在(唯一觸發 fallback) / 1=成功 / 2=格式錯 / 3=重複或例外。Customer=='TSMC' 呼叫 TsmcIeda。

MultiSpecRawData fallback

ReadAndImport/MultiSpecRawData.cs

僅 RawData result==0 時啟用(e30b06c 收斂)。搜尋 test_result_site*_{dbKey}.csv,多 site 共用單一 lots_info。

⚠️ 多 site 部分失敗 → 前段不回滾(partial import)。

Tester

ReadAndImport/Tester.cs

4 表:tester_device_info / tester_status / tester_sw_version / tester_production_analysis

新增 SWT result / SWT respond 欄位;CompareStatus 子集驗證相容新舊 CSV。

FailPin

ReadAndImport/FailPin.cs

表:fail_pin_rate_info / _list / _list_pin_ball / _test_result

SN Num 相容:新 dataStartIndex=4 / 舊=3,HasSnNum 旗標切換。'@' 分隔測試結果。

RecoveryRate

ReadAndImport/RecoveryRate.cs

笛卡兒積展開 → FinalRecoveryRateTable → 寫 recovery_rate 表(check_status 權重 8)。

UiStatus 最簡單

ReadAndImport/UiStatus.cs

單一 ui_status 表,無重複檢查,純同步。獨立於 check_status。

TsmcIeda 唯一固定寬度

ReadAndImport/TsmcIeda.cs

唯一以 Substring 固定寬度解析、唯一純同步、唯一 ListDirectory 批次。lot_mapping.csv → ase_lot(讀入記憶體快取,不寫獨立表);寫 ieda_title / ieda_content。IEDA/CSV/LotID 路徑已收斂到 ImportData / IImportFileSource helper。

9 資料庫存取層

DbAccess.cs 控制表

控制表 db_key / db_key_ui_status;計算 check_status 公式;失敗通知清單由 mail_temp.txt 經 SelectFailDbKeyFromFile() 讀回。

DatabaseService.cs typed-only

ExecuteQuery 回 DbQueryResult;ExecuteCommand 回 DbCommandResult。CheckDatabaseAndTableExists 只檢查存在性、不自動建表。

DBmysql.cs typed-only

每次 new MySqlConnection;MySqlConnectionManager 單例連線字串(池 Min5/Max100)。FilterSqlCommand 為 S2 前殘留防線。

DbObject.cs 回應契約

DbSqlRequest = Query + Parameters;DbQueryResult = JArray Data + Error;DbCommandResult = AffectedRows + InsertId + Error。

⚠️ 錯誤以 Error 字串傳遞,不拋例外:呼叫端必須主動 string.IsNullOrEmpty(response.Error) 檢查。現況是 Dapper 直連 MySQL,沒有外部 HTTP API Service(舊報告所述 API surface 已清理;Task 5 後 DB result surface 為 typed-only)。

10 資料模型

本服務寫入的匯入表與關聯。實線 = FK on delete cascade;虛線 = 僅靠 index / 欄位值的軟關聯(無 FK,刪父表不連帶)。

erDiagram
    db_key ||..o{ lots_info : "db_key 值"
    db_key ||..o{ tester_device_info : "db_key 值"
    db_key ||..o{ fail_pin_rate_info : "db_key 值"
    db_key ||..o{ recovery_rate : "db_key 值"
    lots_info ||--o{ lots_statistic : "FK lot_id cascade"
    lots_info ||--o{ lots_result : "FK lot_id cascade"
    tester_device_info ||--o{ tester_status : "FK cascade"
    tester_device_info ||--o{ tester_production_analysis : "FK cascade"
    tester_device_info ||..o{ tester_sw_version : "index 無FK"
    fail_pin_rate_info ||--o{ fail_pin_rate_list : "FK cascade"
    fail_pin_rate_list ||--o{ fail_pin_rate_list_pin_ball : "FK cascade"
    fail_pin_rate_list ||--o{ fail_pin_rate_test_result : "FK cascade"
    ieda_title ||..o{ ieda_content : "title_id 軟關聯 無FK"
    db_key_ui_status ||..o{ ui_status : "匯入批次對應 無FK"
⚠️ tester_sw_version / ieda_content / ui_status 無 FK cascade,父表刪除不會自動連帶,回滾全靠 app 端 DELETE 順序(見 ⑪)。db_key 是所有匯入根表的邏輯關聯鍵(每張根表帶 db_key 欄回指控制表,非 FK)。

12 接手指南

1️⃣ 先讀三節

三執行緒(⑥)+ CheckStatus(⑧)+ 回滾真相(⑪)是理解系統行為的關鍵。

2️⃣ 環境判定

本機 IP 在 10.16.92.67/68 → Prod,否則 Dev(localhost)。開發機預設走 Dev。

3️⃣ 加新處理器

繼承 ImportData、放 ReadAndImport/;新增 DB 寫入走 DbSqlRequest.Parameters / ExecuteInsert,caller 優先吃 typed results。

4️⃣ 動共用檔前

WriteToLog / FileProcess / DbAccess 扇入極高,先確認所有呼叫端。

5️⃣ Build / test

single net8.0-windows restore/build/test;runtime smoke 需 Windows + MySQL/FTP。

6️⃣ Debug 日誌
